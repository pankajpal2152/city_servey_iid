## Login Integration

### Request

```http
POST /api/auth/api-post-authenticate-user
Content-Type: application/json
```

Preferred body:

```json
[
  {
    "USER_NAME": "sai@yopmail.com",
    "PASSWORD": "Abc@1234"
  }
]
```

Object body also works:

```json
{
  "USER_NAME": "sai@yopmail.com",
  "PASSWORD": "Abc@1234"
}
```

### Successful Response

The response contains:

- `Token`
- `status`
- `response`
- user fields from DB such as `USER_SYS_ID`, `FIRST_NAME`, `EMAIL_ID`,
  `SYSTEM_ROLE_NAME`, `ORGANISATION_SYS_ID`

Example shape:

```json
{
  "Token": "<jwt-token>",
  "status": "true",
  "response": "User login is successful.",
  "USER_SYS_ID": 3,
  "FIRST_NAME": "Sai",
  "EMAIL_ID": "sai@yopmail.com",
  "SYSTEM_ROLE_NAME": "Admin",
  "ORGANISATION_SYS_ID": 1
}
```

### Frontend Login Handling

On success:

- Store `Token` in the frontend auth state.
- Use the token for protected APIs.
- Redirect to dashboard or user management page.

On failure:

- If HTTP 400, show missing field validation.
- If HTTP 401, show invalid credentials.
- If HTTP 500, show server or database error message.

## Token Usage

Protected APIs require:

```http
Authorization: Bearer <Token>
```

Token expiry:

- Backend signs login tokens for `7d`.

Recommended UI behavior:

- If API returns 401, redirect user to login.
- If API returns 403, clear token and redirect user to login.
- Do not send protected API requests without token.

## View User List Integration

### Request

```http
GET /api/user/api-get-view-user-list-info?ITEM=VIEW_ALL
Authorization: Bearer <Token>
```

`ITEM` is optional. If it is omitted, the backend sends `VIEW_ALL` to the stored
procedure.

### Expected Response Shape

The response comes from `USP_GET_USER_LIST_ACTIVITY`. UI should use the
`response` field when `status` is `true`.

```json
{
  "status": "true",
  "response": []
}
```

Use this API to populate user tables, user management screens, and user
dropdowns when needed.

## Add/Update User Integration

### Request

```http
POST /api/user/api-post-add-update-user
Content-Type: application/json
Authorization: Bearer <Token>
```

Body:

```json
{
  "ITEM": "ADD_USER",
  "USER_SYS_ID": 0,
  "SYSTEM_ROLE_SYS_ID": 1,
  "FIRST_NAME": "Sai",
  "LAST_NAME": "Roy",
  "GENDER": "Male",
  "MOBILE_NO": "7894561230",
  "EMAIL_ID": "sai@yopmail.com",
  "ORGANISATION_SYS_ID": "1",
  "PASSWORD": "Abc@1234",
  "CREATED_BY": "1"
}
```

### Field Mapping

| UI Field | Payload Key | Required for Add | Suggested UI Validation |
| --- | --- | --- | --- |
| Action | `ITEM` | Yes | Use `ADD_USER` for create. Confirm update value with backend/DB team. |
| User ID | `USER_SYS_ID` | Yes | `0` for add. Existing user id for update. |
| Role | `SYSTEM_ROLE_SYS_ID` | Yes | Dropdown once role list API is available. |
| First name | `FIRST_NAME` | Yes | Required, trim whitespace. |
| Last name | `LAST_NAME` | Yes | Required, trim whitespace. |
| Gender | `GENDER` | Optional until DB confirms | Dropdown or radio. |
| Mobile number | `MOBILE_NO` | Yes | Digits only, length validation as per product rule. |
| Email | `EMAIL_ID` | Yes | Email format validation. |
| Organization | `ORGANISATION_SYS_ID` | Yes | Dropdown once organization list API is available. |
| Password | `PASSWORD` | Yes for add | Enforce product password policy when confirmed. |
| Created by | `CREATED_BY` | Yes | Use logged-in user's `USER_SYS_ID` when available. |

### Response Handling

Success or business validation responses come from the stored procedure.

Example duplicate email response:

```json
{
  "status": "false",
  "response": "This Email Id Already Exists"
}
```

UI should display the `response` message to the user.

## Add/Update Customer Details Integration

### Request

```http
POST /api/customer/api-post-add-update-customer-details
Content-Type: application/json
Authorization: Bearer <Token>
```

Body:

```json
{
  "ITEM": "ADD",
  "CUSTOMER_TYPE": "Individual",
  "CUSTOMER_NAME": "Ravi",
  "BILLING_ADDRESS": "Kolkata",
  "CREATED_BY": 1,
  "PIC": [
    {
      "FULL_NAME": "Ranjan",
      "CONTACT_NO": "7875585",
      "EMAIL_ID": "ranjan@gmail.com",
      "DEPARTMENT": "Back Office",
      "DESIGNATION": "Employee"
    },
    {
      "FULL_NAME": "Manoj",
      "CONTACT_NO": "7852479",
      "EMAIL_ID": "manoj@gmail.com",
      "DEPARTMENT": "Back Office",
      "DESIGNATION": "Employee"
    }
  ]
}
```

### Customer Field Mapping

| UI Field | Payload Key | Required for Add | Suggested UI Validation |
| --- | --- | --- | --- |
| Action | `ITEM` | Yes | Use `ADD` for create. Confirm update value with backend/DB team. |
| Customer type | `CUSTOMER_TYPE` | Yes | Dropdown or radio, for example `Individual`. |
| Customer name | `CUSTOMER_NAME` | Yes | Required, trim whitespace. |
| Billing address | `BILLING_ADDRESS` | Yes | Required, allow multiline if UI needs. |
| Created by | `CREATED_BY` | Yes | Use logged-in user's `USER_SYS_ID` when available. |
| PIC list | `PIC` | Optional until DB confirms | Array of customer contact persons. |
| PIC full name | `PIC[].FULL_NAME` | For each PIC row | Required when adding a PIC row. |
| PIC contact no | `PIC[].CONTACT_NO` | For each PIC row | Digits/phone validation as per product rule. |
| PIC email | `PIC[].EMAIL_ID` | For each PIC row | Email format validation. |
| PIC department | `PIC[].DEPARTMENT` | For each PIC row | Text/dropdown as per product rule. |
| PIC designation | `PIC[].DESIGNATION` | For each PIC row | Text/dropdown as per product rule. |

The backend expects a root object. If a one-item array is sent accidentally, the
backend unwraps it before calling the stored procedure so `PIC` remains at the
root object path expected by the database.

## JavaScript Fetch Example

```javascript
const baseUrl = import.meta.env.VITE_API_BASE_URL || 'https://api-dev.citysurveyors.com.sg';

export async function login(userName, password) {
  const response = await fetch(`${baseUrl}/api/auth/api-post-authenticate-user`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify([{ USER_NAME: userName, PASSWORD: password }]),
  });

  const data = await response.json();

  if (!response.ok) {
    throw new Error(data.response || data.error || 'Login failed');
  }

  return data;
}

export async function getUserList(token, item = 'VIEW_ALL') {
  const response = await fetch(`${baseUrl}/api/user/api-get-view-user-list-info?ITEM=${encodeURIComponent(item)}`, {
    method: 'GET',
    headers: {
      Authorization: `Bearer ${token}`,
    },
  });

  const data = await response.json();

  if (!response.ok) {
    throw new Error(data.response || data.error || 'User list failed');
  }

  return data;
}
```

## Suggested UI Pages

### Login Page

Fields:

- Email or username
- Password

Behavior:

- Disable submit while loading.
- Show DB `response` message on failure.
- Store token and user details on success.

### User Management Page

Fields:

- Role
- First name
- Last name
- Gender
- Mobile number
- Email
- Organization
- Password

Behavior:

- Require login token.
- Load existing users with the user list API.
- Send payload to add/update API.
- Show success or failure message from `response`.
- For demo, use existing sample email to show duplicate validation safely.

### Customer Management Page

Fields:

- Customer type
- Customer name
- Billing address
- PIC/contact person rows

Behavior:

- Require login token.
- Load existing customers with the customer list API.
- Load one customer profile/details record with the specific customer API when
  opening a customer row.
- Send payload to customer details API.
- Show success or failure message from `response`.
- During demos, use validation-only body `{ "ITEM": "ADD" }` unless the team
  approves creating customer/PIC records in the dev database.

### Project Management Page

Fields:

- Customer
- PIC/contact person
- Site address
- Assigned surveyor
- Type of survey
- Type of property

Behavior:

- Require login token.
- Load customers first so the user can choose `CUSTOMER_SYS_ID` and `PIC_SYS_ID`.
- Load existing projects with the project list API.
- Send payload to project add/update API.
- Show success or failure message from `response`.
- During demos, use validation-only body `{ "ITEM": "ADD" }` unless the team
  approves creating project records in the dev database.

### System Role Management Page

Fields:

- Organization
- System role
- Role description
- User type

Behavior:

- Require login token.
- Load existing system roles with the system-role list API.
- Send payload to system role add/update API.
- Show success or failure message from `response`.
- During demos, use validation-only body `{ "ITEM": "ADD" }` unless the team
  approves creating system role records in the dev database.

- Do not hardcode database credentials in UI code.
